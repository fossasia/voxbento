Voxbento is built on a single guiding principle: the audio path should be as short and as direct as possible. Rather than routing interpreter microphone audio through application servers, Voxbento sends audio straight from the interpreter's browser to MediaMTX — an open-source media server — and from there straight to each listener's browser. The Voxbento coordination portal sits alongside that path to handle coordination, but it never touches the audio itself.

## The Interpretation Pipeline

Here is how audio travels from the speaker on stage to a listener's headphones, step by step:

### Floor audio: the speaker enters Jitsi

The presenter or speaker joins a self-hosted Jitsi Meet session. This session carries the event's floor audio — the source language that interpreters listen to and translate.

    Voxbento embeds the relevant Jitsi session as an iframe directly inside each interpreter's booth page. The interpreter monitors floor audio and video from within the same browser tab where they will broadcast their translation, which keeps the workflow contained and the latency low.
  ### Interpreter audio: mic capture and WebRTC publish

When the interpreter clicks **Go Live**, the browser calls `navigator.mediaDevices.getUserMedia` to capture microphone input. Before the audio leaves the browser, the browser's built-in DSP stack applies:

    - Echo cancellation (to prevent Jitsi floor audio from bleeding into the interpreter's mic)
    - Noise suppression
    - Auto gain control

    The processed audio track is added to an `RTCPeerConnection`. The browser then performs a WHIP handshake — a standard HTTP POST that negotiates the WebRTC session with MediaMTX. After the handshake, audio flows over the established WebRTC media connection directly to MediaMTX.

    :::info
The audio path goes directly from your browser to MediaMTX and then to listeners — Voxbento's coordination portal is only involved in booth state, role checks, and captions, and never relays interpreter microphone data.
:::
  ### MediaMTX: ingest and remux

MediaMTX terminates the WebRTC connection from the interpreter's browser, decodes the audio, and makes it available on a named path that corresponds to the booth's identity:

    ```text
    {event_slug}/{language_code}
    ```

    For example, the English booth at PyCon 2026 occupies the path `pycon2026/en`. The WHIP and WHEP URLs for that booth follow the pattern:

    ```text
    # Interpreter publishes to:
    {mediamtx_base}/pycon2026/en/whip
    
    # Listeners connect to:
    {mediamtx_base}/pycon2026/en/whep
    ```

    MediaMTX holds these paths open even when no interpreter is currently publishing, so listener connections are never dropped between handoffs.
  ### Listener delivery: WHEP (primary) and HLS (fallback)

Listeners connect to the WHEP endpoint for their chosen language. WHEP (WebRTC HTTP Egress Protocol) is the mirror of WHIP — it negotiates a WebRTC session from the server side and streams audio to the listener's browser with sub-second latency.

    For devices that do not handle WebRTC reliably, MediaMTX also produces HLS segments on a separate port. The listener page loads hls.js and falls back to HLS automatically when WHEP is unavailable. HLS introduces roughly three seconds of additional latency but works on virtually any device.

    :::tip
You do not need to configure the fallback manually. The listener page tries WHEP first and switches to HLS transparently if the WebRTC negotiation fails.
:::

Rooms can also define an optional listener-side audio synchronization delay. This is useful when attendees are watching delayed livestream video, delayed captions, embedded conference players, or external media while listening to Voxbento audio. For example, use `1000` ms for a one-second caption delay, `2000` ms for a lightly delayed player, `5000` ms for a typical embedded stream, or `8000` ms for a platform with a larger broadcast buffer. When the room delay is `0` ms, listener playback uses the same direct WHEP-to-HTML-audio path as normal and adds no Web Audio graph or extra latency.

## The Transcription & Translation Pipeline

When live captions or translations are enabled, Voxbento spawns background workers to process the audio in real-time. This pipeline supports both **Booth Audio** (interpreters) and **Floor Audio** (the main conference).

### Floor Audio Capture
To transcribe or translate the original speakers without requiring external integrations, the system launches an isolated **Floor Bot** (`floor-bot`). This headless Chromium instance silently joins the Jitsi meeting, captures the live audio, and pushes it to MediaMTX via RTSP, making it accessible to the transcription workers just like a regular booth.

### Pipeline Steps
1. **Audio Ingest:** The worker uses FFmpeg to pull the live audio stream from MediaMTX via RTSP (`rtsp://mediamtx:8554/{event_slug}/{language_code}` or `rtsp://mediamtx:8554/{event_slug}/floor`).
2. **Transcoding:** FFmpeg transcodes the audio to PCM S16LE (mono, 16kHz or 24kHz depending on the provider) and pipes it to the selected `TranscriptionProvider`.
3. **Speech-to-Text (Transcription):** The provider sends audio to an AI service (OpenAI, Deepgram, ElevenLabs, NVIDIA, or Local Whisper). The `CaptionAggregator` receives raw text, splitting sentences and instantly broadcasting partial updates to listeners (`/ws/captions/{channel_id}`).
4. **Finalization:** Once a full sentence is formed, the aggregator saves it to the database as a canonical `transcript_segment`.
5. **Text-to-Text (Translation):** If target languages are enabled, the Translation Worker detects the finalized segment and spawns parallel requests to LLMs (Anthropic, Groq, Gemini) to translate the sentence. 
6. **Broadcast:** Translations are saved to the database and immediately broadcast via the dedicated WebSocket (`/ws/translations/{channel_id}`).

7. **Text-to-Speech (TTS):** If the room has "Floor Audio TTS (Direct Push)" enabled, the `TTSWorker` instantly receives the finalized translated text and synthesizes speech using the room's configured provider — either **Deepgram Aura** (cloud WebSocket) or **Supertonic** (self-hosted, in-process ONNX). Both emit raw PCM (24 kHz s16le) pushed directly to listeners.
8. **TTS Audio Delivery:** Deepgram returns synthesized speech as raw PCM binary data, which the `tts_manager` broadcasts directly to listeners over a dedicated WebSocket (`/ws/tts/{room_id}`). Listeners hear the translated audio natively via the Web Audio API with sub-second latency, bypassing the WHEP stream entirely.

:::info
Transcription, Translation, and TTS workers are isolated from the main audio path. If an API provider lags, fails, or drops chunks, the core interpretation audio remains perfectly stable and unaffected.
:::

## Booth Identity

Every booth in Voxbento is uniquely identified by two coordinates:

| Coordinate | Format | Example |
| --- | --- | --- |
| `event_slug` | Lowercase alphanumeric and hyphens, 1–64 characters | `pycon2026` |
| `language_code` | ISO 639-1 two-letter code | `en` |

These two values combine to form the **MediaMTX path** (`pycon2026/en`) and the **booth ID** (`pycon2026-en`). Both are derived at runtime — there is no separate stored URL. If you change the event slug or language code, the derived URLs update automatically.

## Interpreter Handoff

In a professional interpretation setting, interpreters rotate every 20–30 minutes to prevent fatigue. Voxbento handles this transition without interrupting listeners.

Here is what happens when a room_coordinator switches the active interpreter:

1. The room_coordinator clicks **Set Active** next to the incoming interpreter in the booth.
2. The portal updates the active interpreter state and notifies all booth participants over WebSocket.
3. The incoming interpreter's browser calls the gated `/api/booth/{booth_id}/whip-url` endpoint, receives the WHIP URL, and begins publishing.
4. MediaMTX is configured with `overridePublisher: yes` — it immediately accepts the new publisher and disconnects the outgoing interpreter's WHIP session.
5. All existing WHEP listeners receive the new publisher's audio within approximately one to two seconds. Their `RTCPeerConnection` stays open throughout; MediaMTX routes the new track into it automatically.
6. **Transcription Continuity:** The backend background transcription worker (FFmpeg) continuously retries pulling from the RTSP stream. Even if the stream drops for a few milliseconds during the handoff, the worker aggressively reconnects, ensuring zero loss of translation pipeline state.

:::info
Listeners on the HLS fallback path experience a longer gap during handoff — roughly 10–15 seconds — while hls.js recovers from the muxer reset that occurs when the publisher changes.
:::

## Roles

Voxbento has three participant roles that are active during a live event:

### Coordinator

Manages the booth. Assigns which interpreter is active, initiates handoffs, and monitors participant state. Coordinators can see the full participant roster and internal booth chat. They cannot publish audio themselves.
  

  ### Interpreter

    Monitors floor audio via the Jitsi iframe and publishes translated audio to MediaMTX via WHIP when they are the active interpreter. Only the active interpreter can go live — all others are held in standby.
  

  ### Listener

    Receives the interpreted audio stream from MediaMTX via WHEP or HLS. Listeners connect to the event's public listener page and choose a language — no login required by default.
  

## Live Captions

When transcription is enabled for a booth, Voxbento pipes the active interpreter's audio to a transcription backend (Whisper, OpenAI, Deepgram, ElevenLabs, or NVIDIA Riva) and broadcasts the resulting text to all listeners in real time.

Captions are delivered over a dedicated WebSocket endpoint:

```text
/ws/captions/{booth_id}
```

For example:

```text
/ws/captions/pycon2026-en
```

Listeners connected to the caption WebSocket receive transcription segments as they are produced. The caption stream is independent of the audio stream — listeners can receive captions even while audio is playing through WHEP or HLS.

## Security Layers

Publishing audio to a booth is protected by three independent enforcement layers:

<Tabs>
  <TabItem value="role-and-active-interpreter-check" label="Role and active-interpreter check">

Voxbento's server-side coordination layer rejects any attempt to activate a microphone or begin audio publishing unless the participant holds the `interpreter` role **and** is the currently designated active interpreter for that booth. Coordinators and listeners cannot publish audio under any circumstances.

</TabItem>
  <TabItem value="whip-url-gating" label="WHIP URL gating">

Before the browser can start publishing, it must request the MediaMTX ingest URL from the Voxbento API (`GET /api/booth/{booth_id}/whip-url`). That URL is returned only to the active interpreter. Any other participant receives HTTP 403 and never receives a usable ingest address.

</TabItem>
  <TabItem value="media-server" label="Media server">

MediaMTX is configured to allow only one publisher per stream path at a time. Even if a stale or unexpected session somehow reaches the media server, it immediately displaces any existing publisher — ensuring a single, authoritative audio source at all times.

</TabItem>
</Tabs>
