The Voxbento API uses bearer token authentication. You call the `voxbento.com` token endpoint once with your booth access token, receive a signed JWT in return, and then include that JWT as an `Authorization: Bearer {token}` header on every subsequent request. Tokens expire after 24 hours by default, at which point you request a fresh one.

## Obtaining a Token

Send a `POST` request to the Voxbento token endpoint with a JSON body containing the `token` field set to your booth access token. The `BOOTH_ACCESS_TOKEN` value is provided by the Voxbento platform.

**Endpoint**

```
POST /api/auth/token
```

**Request body**

```json
{
  "token": "your-booth-access-token"
}
```

**Response**

```json
{
  "access_token": "eyJ...",
  "token_type": "bearer"
}
```

**Example request**

<Tabs>
<TabItem value="curl" label="cURL">

```bash
curl -X POST https://voxbento.com/api/auth/token \
  -H 'Content-Type: application/json' \
  -d '{"token": "your-booth-access-token"}'
```

</TabItem>
<TabItem value="python" label="Python">

```python
import httpx

response = httpx.post(
    "https://voxbento.com/api/auth/token",

```

:::info
The `BOOTH_ACCESS_TOKEN` is provided by the Voxbento platform. Ensure you have the correct credentials before attempting to authenticate.
:::

## Using the Token

Include the token you received as a bearer credential in the `Authorization` header of every subsequent API request.

```
Authorization: Bearer eyJ...
```

**Example — listing booths for an event**

```bash
curl https://voxbento.com/api/events/pycon2026/booths \
  -H 'Authorization: Bearer eyJ...'
```

Store the token in a secure location such as an environment variable or a secrets manager. Do not embed it in client-side code or version-control it.

## Token Lifetime

:::tip
Request a new token proactively before your current one expires to avoid a gap in service. Checking the response status on each request and refreshing on a `401` is a safe fallback strategy.
:::

Tokens expire after 86400 seconds (24 hours) by default. When a token expires, the API returns a `401 Unauthorized` response — simply call `POST /api/auth/token` again to obtain a fresh token and retry the original request.

## Error Responses

| Status | Meaning |
|---|---|
| `401 Unauthorized` | The `Authorization` header is missing, the token is malformed, or the token has expired. Request a new token and retry. |
| `403 Forbidden` | The token is structurally valid but the request targets a resource you are not permitted to access. |

## Cookie-Based Authentication (Internal UI)

While external integrations use the Bearer tokens described above, the Voxbento frontend UI relies on a strict hierarchy of HTTP-only JWT cookies to authenticate users.

### The Token Hierarchy

Voxbento issues three distinct cookies, which are evaluated in the following order of precedence:

1. **`admin_token`**: Issued via the `/admin/login` page using the global `ADMIN_PASSWORD`. Grants blanket `super_admin` permissions across the entire platform.
2. **`user_token`**: Issued when a registered user logs in. It contains claims about their identity (`sub`) and whether they have site-wide admin rights (`is_admin=True`).
3. **`session_token`**: Issued when a user clicks a magic invite link (`/join/{token}`). It is scoped to a specific `event_slug` and `booth_id` with a specific role.

### Role Resolution (`resolve_booth_role`)

When a user joins a booth via WebSocket, the backend `resolve_booth_role` function determines their ultimate permissions by checking the database and comparing their tokens.

Voxbento enforces the following role hierarchy:
`super_admin` > `event_owner` > `room_coordinator` > `interpreter` > `listener`

If a user has multiple conflicting roles (e.g., they clicked an `interpreter` invite link, but they are logged in as an `event_owner` for that event), **the highest privilege wins**. 

### The `require_admin` Dependency

For administrative HTTP routes, Voxbento uses the `require_admin` FastAPI dependency. This dependency:
1. Checks for a valid `admin_token` cookie. If found, access is granted.
2. If missing, checks for a valid `user_token`.
3. If the user token has `is_admin=True`, they are treated as a `super_admin` and granted access.
4. If not a `super_admin`, it checks the database to see if the user is an `event_owner` for the specific `event_id` or `room_id` in the URL path.
5. If neither condition is met, it returns HTTP `403 Forbidden`.
