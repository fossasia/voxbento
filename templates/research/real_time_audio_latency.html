  • Glass-to-Glass Latency: The total time elapsed from the moment an interpreter speaks to the moment the attendee hears the audio.
  • TCP vs UDP: Protocols based on UDP (like WebRTC) avoid head-of-line blocking, essential for real-time delivery.
  • Jitter Buffers: Client-side buffers absorb network instability but artificially increase total latency. Tuning these is critical.
  • Measurement Methodology: Acoustic loopback testing is the only accurate way to measure total system latency, bypassing software clock drift.

    The Anatomy of Latency

    In a WebRTC-based architecture (like VoxBento's integration with MediaMTX), total latency is the sum of several distinct phases:

    Phase Description Typical Range (ms)
    Hardware Capture Microphone ADC, OS audio stack, browser Web Audio API overhead. [Placeholder: 20-50ms]
    Ingest Network (WHIP) Browser encoding (Opus) and UDP transmission to the server. [Placeholder: 50-150ms]
    Server Routing Media server (e.g., MediaMTX) receiving, demuxing, and forwarding packets. [Placeholder: 5-20ms]
    Egress Network (WHEP) Server transmission to the attendee's browser. [Placeholder: 50-150ms]
    Client Jitter Buffer & Playback Browser holding packets to reorder/smooth playback, plus hardware DAC. [Placeholder: 50-200ms]

    Acoustic Loopback Measurement Methodology

    Software-based timestamping is prone to NTP drift and clock synchronization errors. To measure true end-to-end latency:

    1. Generate a sharp, distinct audio impulse (e.g., a metronome click) next to the input microphone.
    2. Capture the input impulse on Channel L of a local audio recorder.
    3. Route the attendee output (WHEP stream) to Channel R of the same local audio recorder.
    4. Analyze the resulting stereo waveform in an audio editor (like Audacity) to measure the exact millisecond delta between the impulses on Channel L and Channel R.

    Impact of Geography

    WebRTC relies heavily on peer-to-peer or client-to-server proximity. A user connecting to a server in Frankfurt from Berlin will experience drastically lower latency than a user connecting to Frankfurt from Sydney. For global events, deploying edge relays or a cascading media server topology is necessary to maintain sub-500ms latency globally.
